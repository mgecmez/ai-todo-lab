# Release Notes — v1.0.0 Sprint 4 (BL-012)

**Release Date:** 2026-03-22
**Type:** Security / Auth
**Scope:** Backend + Mobile Frontend

---

## Overview

Sprint 4 replaces the long-lived 7-day JWT with a short-lived access token (15 min) + long-lived refresh token (30 days) pair. Users stay logged in seamlessly — the app silently refreshes the session in the background without any interruption.

---

## What's New

### Refresh Token System (BL-012)

- **Access token lifetime:** 15 minutes (down from 7 days)
- **Refresh token lifetime:** 30 days, stored as SHA-256 hash in the database
- **Token rotation:** Every refresh call revokes the old token and issues a new pair — a stolen refresh token cannot be reused after one rotation
- **Silent refresh:** When any API call returns `401`, the app automatically calls `/api/auth/refresh`, retries the original request with the new token, and the user never notices
- **Concurrent request handling:** Multiple simultaneous `401` responses queue behind a single in-flight refresh call — no duplicate refresh requests
- **Logout revocation:** Logging out calls `POST /api/auth/logout` to revoke the refresh token in the database, in addition to clearing local storage

### New Backend Endpoints

| Method | Path | Description |
|--------|------|-------------|
| POST | `/api/auth/refresh` | Exchange a valid refresh token for a new token pair |
| POST | `/api/auth/logout` | Revoke the refresh token (requires `Authorization` header) |

Both `/api/auth/refresh` and `/api/auth/register` are now covered by rate limiting (BL-016).

## Breaking Changes

**`AuthResponse` shape changed:** `token` field renamed to `accessToken`; new `refreshToken` field added. Any client consuming the raw API must update accordingly.

---

## Security Notes

- Refresh tokens are stored as SHA-256 hex digests only — the raw token never touches the database
- Expired and revoked tokens return identical `401` responses — no information leakage
- Token rotation means a compromised refresh token is invalidated after first legitimate use
- Refresh endpoint is rate-limited (inherits the `"login"` policy: 5 req / 15 min per IP)
